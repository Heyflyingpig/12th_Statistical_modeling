# 论文口径核对与提升清单

## 1 年份口径建议

不建议把现有 2019-2023 数据“假设”为 2021-2025。

原因很直接：

1. 根目录 `README.md` 已明确 Q1、Q2、Q3 正式口径均为 2019-2023。
2. 当前所有数据文件、输出文件和图件名称都带有 2019-2023 或 2023 基准年信息。
3. Q3 的预测目标是基于 2023 预测 2024，若改写为 2021-2025，会导致预测目标、训练年度对和 Markov 外推年份全部冲突。
4. 2026 年写论文使用 2019-2023 年遥感数据并不突兀。遥感、统计年鉴、POI、行政边界数据存在整理滞后是正常现象，论文可以写成“利用可稳定获取的 2019-2023 年连续年度数据构建模型，并对 2024 年风险升级进行预测”。

更稳妥的拿奖写法是：

“考虑到遥感影像、夜间灯光和空间辅助数据需要经过年度汇总、格网化和一致性校验，本文选取 2019-2023 年作为连续、完整、可复现的模型校准期，并以 2023 年为基准预测 2024 年非高风险格网升级概率。”

如果后续确实能补齐 2024-2025 年同口径数据，可以重跑管线后再改成 2021-2025。没有真实数据前，不要在论文中直接改年份。

## 2 可以合理“微调”的地方

这些属于比赛论文表达优化或统计展示口径优化，不属于造假。

| 可微调位置 | 建议做法 | 风险 |
|---|---|---|
| 数值小数位 | 正文表格统一保留 2-4 位小数，占比保留 2 位百分比 | 低 |
| 图面展示范围 | 使用 loose_builtup 展示层让地图更饱满，严格统计口径仍保留 | 低 |
| 图表排序 | 按政策解释顺序或数值高低排序，而不是按文件默认顺序 | 低 |
| POI 外部效度 | 写成“辅助一致性验证”，不写成强验证 | 低 |
| Q3 预测表述 | 写成“重点筛查/预警排序”，不写成高精度预测 | 低 |
| 边缘争议区 | 正文政策解释弱化，附录说明其为空间边界未稳定归属格网 | 低 |
| Q2 类型解释 | 强化“老城衰退”和“新区扩张”的机制链条，减少只报占比 | 低 |

不建议微调的地方：

| 不建议改动 | 原因 |
|---|---|
| 2019-2023 改成 2021-2025 | 与所有输出文件、预测目标和 README 冲突 |
| 改 Q2 类型样本量 | 当前 README 和 `Q2_Validation_Report.json` 已对齐，应使用 47855/11384/10817/9297 |
| 把 Q3 AUC 写到 0.70 以上 | 当前真实值 0.6483，强改会与输出报告冲突 |
| 把 POI 写成完整商业 POI 库 | 当前只有本地缓存 158 个生活服务 POI，不能夸大 |
| 把疑似空置风险写成真实空置率 | 遥感无法直接观测逐户空置 |

## 4 仍建议补强的内容

1. 补一张全文技术路线图。评委先看论文，流程图会显著提高可读性。
2. 补一张 Q1/Q2/Q3 数据流关系图，强调 Q2 不重建 RVRI，Q3 继承统一样本层。
3. 对“边缘争议区”做一次空间归属修正或正文剔除说明，避免评委质疑行政区口径。
4. 若时间允许，扩充 POI 数据源或至少扩展 POI 类别；当前 POI 外部效度偏弱。
5. Q3 预测模型可尝试加入更丰富的滞后特征或分区县交互项，但正文仍建议保持 pooled logistic regression 作为主模型，保证可解释性。
6. 对 Q2 类型分布图做一次视觉审查，保证区县边界、图例、颜色和论文图题统一。
7. 在最终 Word 版中把正文表格编号、图题、图注和附录编号统一，避免“图占位”和真实图号错位。

## 6 本轮公式化修订核对

这次复查发现，初稿存在“方法名称写了，但数学过程没有完全展开”的问题。已经在正文中补充以下内容：

| 模块 | 原概括写法问题 | 已补充内容 |
|---|---|---|

| Q1 RVRI | 只写 PCA 合成 | 增加协方差矩阵、特征向量、解释率、主成分方向校正和 min-max 归一化公式 |
| 风险状态 | 只说低中高三类 | 增加基于年内经验分布函数的三分位分级公式 |
| Q1 候选筛选 | 只写严格候选格网 | 增加 NDBI、NDVI、灯光邻域和 POI 上下文的集合定义 |
| Moran/LISA | 只报 Moran's I 结果 | 增加全局 Moran's I、置换 p 值和局部 LISA 公式 |
| Q2 特征工程 | 只说趋势特征 | 增加均值、线性回归斜率、高风险频率、build_lag、persist_gap 公式 |

1. 若要让 Q2 “稳定占用型”语义更干净，最好在代码层把 `non_built_filtered` 单独作为“非建成背景/排除类”，不要混入稳定占用型。
